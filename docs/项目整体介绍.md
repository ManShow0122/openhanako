## 1. 项目定位与设计理念

### 核心定位

OpenHanako 是一个**面向普通用户的桌面端 AI Agent**，目标是弥合大多数人与 AI Agent 之间的使用门槛。区别于 Claude Code、Codex 等面向开发者的 CLI 工具，Hanako 提供了完整的图形界面，并针对日常办公场景做了大量优化。

### 设计哲学

- **"像人"而非"像工具"**：每个 Agent 有独立的人格（identity）、能力定义（yuan）、意识（ishiki），通过三层 Prompt 组装塑造独特性格。
- **记忆自然化**：记忆系统设计为"空气"——影响 Agent 的角度和判断，但不在对话中显式提及。近期记忆清晰，远期自然淡化。
- **多 Agent 协作**：多个 Agent 可以通过频道（Channel）群聊协作，或通过 DM 互相委派任务，各自有独立的记忆和人格。
- **安全优先**：双层沙盒隔离（应用层 PathGuard + OS 层 Seatbelt/Bubblewrap），Agent 权限在用户掌控之中。
- **开放生态**：兼容 Skills 社区生态，Agent 可从 GitHub 安装技能，也可自主编写新技能。

### 关键架构特点

1. **Engine 是门面（Facade）**：`HanaEngine` 不直接实现业务逻辑，而是将所有功能委托给 5 个专职 Manager，自身只做代理转发和生命周期协调。
2. **Hub 是调度中枢**：`Hub` 包装 Engine，统一处理所有消息路由（桌面端 owner、Bridge guest/owner、隔离执行），并管理后台调度（心跳、Cron、频道路由）。
3. **同进程架构**：Hub 和 Engine 在同一 Node 进程中运行，`hub.send()` 直接调用 Engine 方法，无 IPC 开销。
4. **Electron 通过 fork 启动 Server**：桌面端主进程 fork 出 server 子进程，通过 stdio 传递端口和 token，前端通过 HTTP/WebSocket 与 server 通信。

## 4. 核心模块详解

### 4.1 HanaEngine（`core/engine.js`）

Engine 是整个系统的**门面层（Facade）**，不直接实现业务逻辑，通过代理方法将调用转发给 5 个专职 Manager。

**内部组成：**

| Manager | 职责 |
|---------|------|
| `AgentManager` | 多 Agent 扫描/初始化/创建/切换/删除 |
| `SessionCoordinator` | Session 生命周期、隔离执行 |
| `ModelManager` | 模型发现、切换、凭证解析 |
| `ConfigCoordinator` | 配置读写协调、模型同步 |
| `BridgeSessionManager` | 外部平台 Bridge session 管理 |
| `SkillManager` | Skills 加载、同步、热更新 |
| `PreferencesManager` | 全局偏好（沙盒、语言、时区等） |
| `ChannelManager` | 频道文件管理 |

**初始化顺序（`engine.init()`）：**
1. Provider 迁移（旧配置格式升级）
2. 所有 Agent 初始化（焦点 Agent 先行，其余并行）
3. Pi SDK 初始化（AuthStorage + ModelRegistry）
4. ResourceLoader + SkillManager 初始化
5. 模型发现与默认模型设置
6. 收藏模型迁移（一次性）
7. Skills 同步 + 文件监听
8. Bridge 孤儿清理

**关键静态属性：**
- `PATROL_TOOLS_DEFAULT`：心跳/Cron 任务允许使用的工具白名单

### 4.2 Agent（`core/agent.js`）

每个 Agent 实例代表一个独立的 AI 助手，包含：

**人格三层结构：**
```
identity.md    → 身份定义（我是谁）
yuan/xxx.md    → 能力定义（我能做什么）
ishiki.md      → 意识定义（我怎么思考）
```

**System Prompt 组装顺序：**
1. 人格三层（identity + yuan + ishiki）
2. 用户档案（`user.md`）
3. 记忆使用规则（如果记忆开启）
4. 置顶记忆（`pinned.md`）
5. 普通记忆（`memory.md`）
6. Skills 注入（`formatSkillsForPrompt`）
7. 文件呈现规则、Artifact 规则、浏览器使用规则
8. 主动技能获取引导（如果开启）
9. 书桌路径说明
10. 当前日期时间

**记忆开关层级：**
- `memoryMasterEnabled`：Agent 级总开关（config.yaml）
- `memorySessionEnabled`：Session 级开关（session-meta.json）
- 两者都开启，记忆才注入 System Prompt

**内置工具列表：**
- 记忆工具：`search_memory`、`pin_memory`、`unpin_memory`、`recall_experience`、`record_experience`
- 网络工具：`web_search`、`web_fetch`
- 系统工具：`todo`、`cron`、`notify`、`present_files`、`create_artifact`
- 协作工具：`channel`（频道发消息）、`ask_agent`、`dm`（私信）、`delegate`（委派任务）
- 其他：`browser`、`install_skill`

### 4.3 AgentManager（`core/agent-manager.js`）

**Agent 目录结构（`~/.hanako/agents/<agentId>/`）：**
```
config.yaml         # Agent 配置（模型、人格类型、记忆开关等）
identity.md         # 自定义身份（覆盖模板）
ishiki.md           # 自定义意识（覆盖模板）
public-ishiki.md    # 对外意识（Bridge guest 模式使用）
pinned.md           # 置顶记忆
channels.md         # 频道订阅配置
memory/
  memory.md         # 主记忆文件
  facts.db          # FactStore（SQLite）
sessions/           # 对话 Session 文件（JSONL）
  session-titles.json
  session-meta.json
bridge/             # Bridge session 文件
  bridge-sessions.json
  owner/
  guests/
activity/           # 心跳/Cron 活动记录
  activities.json
desk/               # 书桌系统
  jian-registry.json
  overwatch.md
  cron-jobs.json
  cron-runs/
avatars/            # 头像文件
learned-skills/     # Agent 自主学习的 Skills
```

**Agent 切换流程：**
1. 暂停 Heartbeat（Cron 不中断）
2. 清理当前所有 Session
3. 切换 `activeAgentId`
4. 更新默认模型
5. 恢复 Heartbeat + 重新注入 Handler
6. 同步 Skills
7. 创建新 Session

**Agent 列表缓存：** 30 秒 TTL，增删改时主动失效。

### 4.4 SessionCoordinator（`core/session-coordinator.js`）

**Session 类型：**

| 类型 | 说明 | 持久化 |
|------|------|--------|
| 普通 Session | 桌面端主对话 | 是（JSONL） |
| Bridge Session | 外部平台对话 | 是（bridge/ 目录） |
| Isolated Session | 心跳/Cron/频道 | 否（执行完删除） |
| Activity Session | 可提升为普通 Session | 是（activity/ 目录） |

**隔离执行（`executeIsolated`）：**
- 用于心跳巡检、Cron 任务、频道回复
- 工具白名单过滤（`PATROL_TOOLS_DEFAULT`）
- 浏览器自动切换为 headless 模式
- 支持 `AbortSignal` 中止
- 执行完毕后自动删除临时 Session 文件

**Session 缓存：** 最多缓存 20 个 Session，LRU 淘汰。

### 4.5 ModelManager（`core/model-manager.js`）

**四类模型配置：**

| 模型字段 | 用途 | 配置位置（preferences key） |
|---------|------|--------------------------|
| `models.chat` | 主对话模型 | agent config.yaml |
| `models.utility` | 轻量任务（摘要、频道 triage） | 全局 `utility_model` |
| `models.utility_large` | 深度分析（记忆编译、日记、频道 triage 大模型） | 全局 `utility_large_model` |
| `models.summarizer` | 滚动摘要专用（可选，缺省复用 utility） | 全局 `summarizer_model` |
| `models.compiler` | 记忆编译专用（可选，缺省复用 utility_large） | 全局 `compiler_model` |

> 后四项通过 `ConfigCoordinator.getSharedModels()` 读取，存储在 `preferences.json` 中，跨所有 Agent 共享。

**模型状态：**
- `defaultModel`：设置页面选择的，持久化，Bridge 使用
- `sessionModel`（`currentModel`）：聊天页面临时切换，只影响桌面端当前 Session

**凭证解析优先级：** 全局 `providers.yaml` → agent `config.yaml` providers 块 → `auth.json`（OAuth）

### 4.6 Hub（`hub/index.js`）

**消息路由优先级（按顺序匹配）：**

```
1. Agent → Agent 私信（from + to 都有）
2. 桌面端 owner（无 sessionKey，无 ephemeral，role=owner）
3. Bridge guest（有 sessionKey，role=guest）
4. Bridge owner（有 sessionKey，无 ephemeral）
5. 隔离执行（ephemeral=true，用于 cron/heartbeat/channel）
```

**Hub 子模块：**
- `EventBus`：统一事件总线，替代旧的 engine.subscribe
- `Scheduler`：Heartbeat（只跑当前 active agent）+ Cron（所有 agent 并发）
- `ChannelRouter`：频道 triage + 两轮 Agent Session 回复
- `GuestHandler`：外部平台访客消息处理（消息前缀标注发送者身份 `[来自 xxx]`，注入群聊/访客上下文标签，使用 `public-ishiki` 替代完整 System Prompt，不暴露主人隐私）
- `AgentMessenger`：Agent 间私信
- `DmRouter`：DM 消息路由

### 4.7 Scheduler（`hub/scheduler.js`）

**Heartbeat（心跳巡检）：**
- 只跑当前 active agent
- 定期检查书桌文件变化（`jian-registry.json` 追踪）
- 检测到"笺"（Jian）文件时触发专项巡检
- Agent 切换时重建 Heartbeat，Cron 不中断

**Cron（定时任务）：**
- 所有 agent 独立并发运行，不随 active agent 切换中断
- 同一 agent 同时只运行一个 Cron（防并发写冲突）
- 执行结果写入 `ActivityStore`，通过 EventBus 广播

### 4.8 ChannelRouter（`hub/channel-router.js`）

**频道回复两阶段流程：**

```
阶段 1 - Triage（判断是否回复）：
  - 如果消息中有 @agentName，直接回复
  - 否则调用 utility_large 模型判断（YES/NO）
  - 超时 10 秒

阶段 2 - 两轮 Agent Session 生成回复：
  第一轮：读取频道消息 + 查阅记忆（内部思考，不输出）
  第二轮：生成实际回复（直接发送到频道）
  - 默认 30 字以内，话题需要时可到 1000 字
  - 回复 [NO_REPLY] 则不发送
```

**频道记忆摘要：** 对话结束后用 utility 模型生成摘要，写入 agent 的 FactStore。

## 5. 数据流与通信机制

### 5.1 Electron 主进程 ↔ Server 子进程

```
Electron main.cjs
  └─ fork(server/boot.cjs)
       ├─ stdio IPC: { type: "ready", port, token }  → 主进程
       └─ stdio IPC: { type: "shutdown" }             ← 主进程
```

Server 启动后将 `{ pid, port, token }` 写入 `~/.hanako/server-info.json`，供复用检测。

### 5.2 前端 ↔ Server

- **HTTP REST API**：所有管理操作（Agent CRUD、配置、Skills 等）
- **WebSocket**：聊天流式输出（`/ws/chat`）
- **认证**：所有请求需携带 `Authorization: Bearer <token>`（token 随机生成，写入 server-info.json）

### 5.3 事件系统

```
EventBus.emit(event, sessionPath)
  ↓
所有订阅者（WebSocket 推送、桌面端 IPC 等）

事件类型：
  message_update      - 消息流式更新
  tool_execution_*    - 工具执行状态
  activity_update     - 心跳/Cron 活动记录
  cron_job_done       - Cron 任务完成
  channel_new_message - 频道新消息
  notification        - 系统通知
  devlog              - 开发日志
  browser_bg_status   - 浏览器后台状态
```

### 5.4 Bridge（外部平台接入）

```
外部平台（Telegram/飞书/QQ）
  ↓ 消息
BridgeManager（lib/bridge/bridge-manager.js）
  ↓
Hub.send({ sessionKey, role: "guest"|"owner", meta })
  ↓
BridgeSessionManager.executeExternalMessage()
  ↓ 持久化 session（bridge/owner/ 或 bridge/guests/）
  ↓ 捕获回复文本
  ↓ onDelta 回调（流式推送）
外部平台（回复）
```

**Bridge session 索引：** `bridge-sessions.json`，记录 sessionKey → 文件路径 + 用户元数据。

## 6. 功能系统详解

### 6.1 记忆系统

**三层记忆架构：**

```
置顶记忆（pinned.md）
  - 用户主动要求记住的内容
  - 永久保留，不自动淡化

普通记忆（memory.md）
  - 从对话中自动提取
  - 近期清晰，远期淡化

经验记录（FactStore / facts.db）
  - 结构化事实存储（SQLite）
  - 支持标签检索
  - 频道对话摘要也写入此处
```

**记忆更新触发：**
- `MemoryTicker.notifyTurn(sessionPath)`：每轮对话后触发
- `MemoryTicker.notifySessionEnd(sessionPath)`：Session 结束时触发 final pass
- `SummaryManager`：负责滚动摘要生成

**记忆注入规则（System Prompt 中）：**
- 记忆像"空气"，影响角度和判断，但不在文字中显式提及
- 禁止使用"我记得"、"你之前说过"等表述
- 当前对话信息优先于记忆

### 6.2 Skills 系统

**Skills 加载路径：**
1. `~/.hanako/skills/`（用户安装的 Skills）
2. `skills2set/`（项目根目录内置 Skills，打包时随应用分发）
3. Agent 的 `learned-skills/`（Agent 自主学习的）

**隐藏 Skills（不对用户展示）：**
- `canvas-design`、`skill-creator`、`skills-translate-temp`

**Skills 热更新：** `SkillManager.watch()` 监听 skillsDir 变化，自动重载并同步到所有 Agent。

**主动技能获取（`learn_skills.allow_github_fetch` 开启时）：**
- Agent 遇到专业任务时主动搜索 `clawhub.ai` 或 GitHub
- 通过 `install_skill` 工具安装，安装后立即应用

### 6.3 书桌系统（Desk）

**书桌 = Agent 的工作目录（`homeCwd`）**，不是系统桌面。

**笺（Jian）：** 类似便签的特殊文件，Agent 会主动读取并执行其中的指令。

**心跳巡检流程：**
1. 定期扫描书桌文件变化
2. 对比 `jian-registry.json` 中的上次状态
3. 发现新文件/变化时构建 prompt 触发隔离执行
4. 执行结果写入 ActivityStore

**书桌 API（`server/routes/desk.js`）：**
- 文件列表、上传、删除、重命名
- 活动记录（ActivityStore）查询
- 心跳手动触发
- Cron 任务 CRUD

### 6.4 沙盒安全

**双层隔离：**

```
应用层 PathGuard（lib/sandbox/）：
  四级访问模式：
  - full-access：无限制（关闭沙盒时）
  - standard：工作目录 + 用户文件（默认）
  - restricted：仅工作目录
  - read-only：只读

OS 层：
  - macOS：Seatbelt（sandbox-exec）
  - Linux：Bubblewrap（bwrap）
```

**沙盒开关：** 偏好设置中 `sandbox` 字段，默认开启。隐藏彩蛋：在"关于"页面点击 HANAKO 图标 5 次可解锁完全访问权限。

### 6.5 多 Agent 协作

**频道（Channel）群聊：**
- 频道文件存储在 `~/.hanako/channels/<channelName>.md`
- 所有订阅了该频道的 Agent（有 `channels.md` 文件）都会参与 triage
- 30 秒 TTL 缓存 Agent 顺序列表

**Agent 间私信（DM）：**
- 通过 `dm` 工具发送
- `DmRouter` 处理路由，触发目标 Agent 的隔离执行

**任务委派（Delegate）：**
- 通过 `delegate` 工具将任务委派给其他 Agent
- 目标 Agent 以隔离执行方式处理

### 6.6 Plan Mode（只读探索模式）

开启后 Agent 只能使用只读内置工具，不能写入或执行命令，适合安全探索场景。通过 `engine.setPlanMode()` 切换。

**只读内置工具白名单（`READ_ONLY_BUILTIN_TOOLS`）：** `["read", "grep", "find", "ls"]`

Plan Mode 开启时，Session 的 active tools 被限制为上述 4 个内置工具 + Agent 的所有自定义工具（`agent.tools`，即记忆、搜索等）。关闭时恢复全部内置工具。

## 11. 关键设计决策与约束

### 11.1 ESM 全栈

项目全部使用 ES Modules（`"type": "module"`），Electron 主进程例外（`main.cjs`、`preload.cjs`、`boot.cjs` 使用 CJS）。混用时注意动态 `import()` 和 `require()` 的区别。

### 11.2 Pi SDK 依赖

Agent 运行时深度依赖 `@mariozechner/pi-coding-agent`，包括：
- `createAgentSession`：创建 Agent 会话
- `SessionManager`：Session 文件管理（JSONL 格式）
- `ModelRegistry`：模型注册与发现
- `AuthStorage`：OAuth 凭证存储
- `SettingsManager`：Session 压缩配置

升级 Pi SDK 时需注意 API 兼容性。

### 11.3 同步 SQLite

`better-sqlite3` 使用同步 API，在 Node.js 单线程中直接调用，无需 async/await。WAL 模式支持并发读。

### 11.4 Agent 切换时的状态清理

切换 Agent 时必须：
1. 暂停 Heartbeat（Cron 不中断）
2. 清理所有 Session（abort streaming + unsub）
3. 清除配置缓存（`clearConfigCache()`）
4. 更新默认模型

违反此顺序可能导致旧 Agent 的 System Prompt 泄漏到新 Session。

### 11.5 Bridge Session 的模型选择

Bridge session 使用 agent 自身 `config.yaml` 中配置的 `models.chat`，而非 Engine 的 `defaultModel`（后者可能被桌面端临时切换）。这确保了 Bridge 行为的一致性。

### 11.6 隔离执行的浏览器 headless

`executeIsolated` 执行期间，浏览器自动切换为 headless 模式（通过引用计数），避免后台任务弹出浏览器窗口干扰用户。执行完毕后恢复原状态。

### 11.7 前端迁移状态

前端正在从 vanilla JS 迁移到 React 19 + TypeScript，迁移尚未完成。新功能应使用 React 开发，旧页面（`onboarding.js`、`editor-window-fallback.js` 等）暂时保留。

### 11.8 Windows 特殊处理

- 需要便携版 Git（`scripts/download-git-portable.js`）
- 打包时额外打包 `vendor/git-portable` 到 Resources
- 沙盒使用不同实现（无 Seatbelt/Bubblewrap）
- 进程退出信号额外监听 `SIGBREAK`
